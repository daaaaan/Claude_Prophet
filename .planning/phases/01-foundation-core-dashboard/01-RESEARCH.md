# Phase 1: Foundation & Core Dashboard - Research

**Researched:** 2026-02-11
**Domain:** Go WebSocket server, Alpine.js frontend, Tailwind CSS styling, go:embed static assets
**Confidence:** HIGH

## Summary

Phase 1 builds a real-time dashboard for the existing Go trading bot. The bot already has a Gin HTTP server (port 4534), SQLite database via GORM, and REST API endpoints for account, positions, orders, and activity data. The dashboard adds a WebSocket layer that polls these existing services and pushes JSON snapshots to browser clients, plus a lightweight frontend using Alpine.js and Tailwind CSS served from embedded static files.

The core architecture is straightforward: a WebSocket Hub (gorilla/websocket v1.5.3) manages browser connections, a Ticker goroutine polls existing services at intervals and publishes events, and the frontend uses Alpine.js stores for reactive state management driven by WebSocket messages. The dashboard package must be isolated so panics cannot crash the trading bot. SQLite WAL mode must be enabled to prevent lock contention between the bot's writes and dashboard reads.

**Primary recommendation:** Use the gorilla/websocket chat example Hub/Client pattern almost verbatim, Alpine.js v3 with `Alpine.store()` for global WebSocket state, Tailwind CSS v4 standalone CLI with `@source` directive for Go template scanning, and `go:embed` for single-binary deployment.

### Pattern 1: WebSocket Hub (Broadcast-Only)
**What:** Central hub manages client connections; dashboard is read-only (no client-to-server messages beyond ping/pong)
**When to use:** Always for this dashboard -- clients only receive data, never send commands
**Example:**
```go
// Source: gorilla/websocket chat example (adapted for broadcast-only)
type Hub struct {
    clients    map[*Client]bool
    broadcast  chan []byte
    register   chan *Client
    unregister chan *Client
}

func (h *Hub) Run() {
    for {
        select {
        case client := <-h.register:
            h.clients[client] = true
        case client := <-h.unregister:
            if _, ok := h.clients[client]; ok {
                delete(h.clients, client)
                close(client.send)
            }
        case message := <-h.broadcast:
            for client := range h.clients {
                select {
                case client.send <- message:
                default:
                    close(client.send)
                    delete(h.clients, client)
                }
            }
        }
    }
}
```

### Pattern 2: Client with Ping/Pong Lifecycle
**What:** Each WebSocket client gets readPump + writePump goroutines with automatic ping/pong heartbeat
**When to use:** Every WebSocket connection -- required for detecting dead clients
**Example:**
```go
// Source: gorilla/websocket chat example
const (
    writeWait  = 10 * time.Second
    pongWait   = 60 * time.Second
    pingPeriod = (pongWait * 9) / 10  // 54 seconds
)

type Client struct {
    hub  *Hub
    conn *websocket.Conn
    send chan []byte  // buffered channel, capacity 256
}

func (c *Client) writePump() {
    ticker := time.NewTicker(pingPeriod)
    defer func() {
        ticker.Stop()
        c.conn.Close()
    }()
    for {
        select {
        case message, ok := <-c.send:
            c.conn.SetWriteDeadline(time.Now().Add(writeWait))
            if !ok {
                c.conn.WriteMessage(websocket.CloseMessage, []byte{})
                return
            }
            c.conn.WriteMessage(websocket.TextMessage, message)
        case <-ticker.C:
            c.conn.SetWriteDeadline(time.Now().Add(writeWait))
            if err := c.conn.WriteMessage(websocket.PingMessage, nil); err != nil {
                return
            }
        }
    }
}

func (c *Client) readPump() {
    defer func() {
        c.hub.unregister <- c
        c.conn.Close()
    }()
    c.conn.SetReadLimit(512)
    c.conn.SetReadDeadline(time.Now().Add(pongWait))
    c.conn.SetPongHandler(func(string) error {
        c.conn.SetReadDeadline(time.Now().Add(pongWait))
        return nil
    })
    for {
        // Read loop exists only to handle pong and detect disconnect
        _, _, err := c.conn.ReadMessage()
        if err != nil {
            break
        }
    }
}
```

### Pattern 3: Dashboard Ticker (Poll + Publish)
**What:** Background goroutine polls existing services at intervals, marshals JSON, broadcasts to Hub
**When to use:** Core data pipeline -- replaces direct REST polling from browser
**Example:**
```go
type Ticker struct {
    hub            *Hub
    tradingService interfaces.TradingService
    activityLogger *services.ActivityLogger
    interval       time.Duration
}

func (t *Ticker) Run(ctx context.Context) {
    ticker := time.NewTicker(t.interval)
    defer ticker.Stop()

    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            t.publishSnapshot(ctx)
        }
    }
}

func (t *Ticker) publishSnapshot(ctx context.Context) {
    // Fetch from existing services (same ones bot uses)
    account, _ := t.tradingService.GetAccount(ctx)
    positions, _ := t.tradingService.GetPositions(ctx)
    activity, _ := t.activityLogger.GetCurrentLog()

    snapshot := DashboardSnapshot{
        Type:      "snapshot",
        Timestamp: time.Now(),
        Account:   account,
        Positions: positions,
        Activity:  activity,
    }

    data, _ := json.Marshal(snapshot)
    t.hub.broadcast <- data
}
```

### Pattern 4: Alpine.js Store for WebSocket State
**What:** Global Alpine store holds all dashboard data; WebSocket updates store; components read from store
**When to use:** All dashboard components -- single source of truth for reactive state
**Example:**
```javascript
// Source: Alpine.js store docs + WebSocket integration pattern
document.addEventListener('alpine:init', () => {
    Alpine.store('dashboard', {
        account: { cash: 0, equity: 0, buying_power: 0 },
        positions: [],
        activities: [],
        connected: false,
        lastUpdate: null,

        init() {
            this.connectWebSocket();
        },

        connectWebSocket() {
            const protocol = location.protocol === 'https:' ? 'wss:' : 'ws:';
            const ws = new WebSocket(`${protocol}//${location.host}/ws`);

            ws.onopen = () => { this.connected = true; };
            ws.onclose = () => {
                this.connected = false;
                setTimeout(() => this.connectWebSocket(), 3000);
            };
            ws.onmessage = (event) => {
                const data = JSON.parse(event.data);
                this.lastUpdate = new Date();
                if (data.type === 'snapshot') {
                    this.account = data.account;
                    this.positions = data.positions;
                    this.activities = data.activities;
                }
            };
        },

        get connectionStatus() {
            if (!this.connected) return 'disconnected';
            if (!this.lastUpdate) return 'connecting';
            const age = (Date.now() - this.lastUpdate.getTime()) / 1000;
            if (age < 5) return 'live';
            if (age < 30) return 'stale';
            return 'disconnected';
        }
    });
});
```

### Pattern 5: Gin + go:embed Integration
**What:** Embed all static assets and templates in Go binary; serve via Gin routes
**When to use:** Production deployment (UI-03 requirement)
**Example:**
```go
package dashboard

import (
    "embed"
    "html/template"
    "io/fs"
    "net/http"

    "github.com/gin-gonic/gin"
    "github.com/gorilla/websocket"
)

//go:embed static/templates static/css static/js
var embeddedFiles embed.FS

var upgrader = websocket.Upgrader{
    ReadBufferSize:  1024,
    WriteBufferSize: 1024,
    CheckOrigin: func(r *http.Request) bool { return true },
}

func Register(router *gin.Engine, hub *Hub) {
    // Parse templates from embedded FS
    templ := template.Must(template.New("").ParseFS(embeddedFiles,
        "static/templates/*.html"))
    router.SetHTMLTemplate(templ)

    // Serve static assets (CSS, JS) from embedded FS
    staticFS, _ := fs.Sub(embeddedFiles, "static")
    router.StaticFS("/dashboard/static", http.FS(staticFS))

    // Dashboard page
    router.GET("/dashboard", func(c *gin.Context) {
        c.HTML(http.StatusOK, "dashboard.html", gin.H{})
    })

    // WebSocket endpoint
    router.GET("/ws", func(c *gin.Context) {
        serveWs(hub, c.Writer, c.Request)
    })
}

func serveWs(hub *Hub, w http.ResponseWriter, r *http.Request) {
    conn, err := upgrader.Upgrade(w, r, nil)
    if err != nil {
        return
    }
    client := &Client{hub: hub, conn: conn, send: make(chan []byte, 256)}
    client.hub.register <- client
    go client.writePump()
    go client.readPump()
}
```

### Pattern 6: Dashboard Package Isolation (WS-05)
**What:** All dashboard code in separate `dashboard/` package with panic recovery
**When to use:** Always -- dashboard crash must not take down trading bot
**Example:**
```go
// In cmd/bot/main.go -- register dashboard with recovery
func main() {
    // ... existing setup ...

    hub := dashboard.NewHub()
    go hub.Run()

    ticker := dashboard.NewTicker(hub, tradingService, activityLogger, 2*time.Second)
    go ticker.Run(ctx)

    // Register routes (panic in handler recovered by Gin's default middleware)
    dashboard.Register(router, hub)

    // ... existing server start ...
}
```
Gin's default `Recovery()` middleware (included in `gin.Default()`) catches panics in HTTP handlers. The Hub and Ticker run in separate goroutines with their own `defer recover()` wrappers.

### Anti-Patterns to Avoid
- **Polling from browser:** Do NOT have the frontend make REST API calls on a timer. Use WebSocket push exclusively (RT-01).
- **One goroutine per data source:** Do NOT create separate tickers for account, positions, and activity. Use a single Ticker that assembles a complete snapshot to avoid partial state on the client.
- **WebSocket in main package:** Do NOT put WebSocket code directly in `cmd/bot/main.go`. Isolate in `dashboard/` package (WS-05).
- **Dynamic Tailwind classes:** Do NOT construct class names dynamically (e.g., `text-${color}-600`). Always use complete static class names. Tailwind scans source files as plain text.
- **Storing WebSocket connection in x-data:** Do NOT put the WebSocket connection inside component-level `x-data`. Use `Alpine.store()` for global state so all components share one connection.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| WebSocket connection management | Custom TCP handling | gorilla/websocket Hub/Client pattern | Handles ping/pong, dead client detection, write deadlines, concurrent writes safely |
| CSS utility framework | Custom CSS classes | Tailwind CSS v4 standalone CLI | Responsive, dark mode, consistent design tokens out of the box |
| Frontend reactivity | Manual DOM manipulation | Alpine.js with `Alpine.store()` | Reactive data binding, no build step, 15kb CDN include |
| WebSocket reconnection | Custom retry logic | Reconnect in `onclose` with exponential backoff | Browser WebSocket API handles this simply; Alpine store manages state |
| Static asset embedding | Build scripts or Docker COPY | `go:embed` directive | Built into Go, zero external tooling, works with `go build` |
| JSON serialization for events | Custom wire format | `encoding/json` with struct tags | Standard, debuggable, Alpine parses with `JSON.parse()` |

**Key insight:** The entire WebSocket infrastructure is well-solved by gorilla/websocket's chat example. The innovation here is in the Ticker (what data to poll, how to structure snapshots) and the frontend layout, not in the plumbing.

## Common Pitfalls

### Pitfall 1: SQLite Lock Contention (CRITICAL)
**What goes wrong:** Bot writes to SQLite while dashboard reads cause "database is locked" errors
**Why it happens:** Default SQLite journal mode (DELETE) allows only one writer OR one reader at a time
**How to avoid:** Enable WAL mode via DSN parameter: `sqlite.Open("file:./data/prophet_trader.db?_journal_mode=WAL")`
**Warning signs:** Intermittent "database is locked" errors in logs, dashboard showing stale data
**Implementation detail:** The existing `database/storage.go` NewLocalStorage function uses `sqlite.Open(dbPath)` -- must be changed to include `?_journal_mode=WAL` in the DSN. WAL mode is persistent once set, survives restarts.

### Pitfall 2: Concurrent WebSocket Writes
**What goes wrong:** Multiple goroutines write to the same WebSocket connection simultaneously, causing panics
**Why it happens:** gorilla/websocket connections are not safe for concurrent writes
**How to avoid:** Use the Client pattern with a buffered `send` channel -- only writePump goroutine writes to conn
**Warning signs:** Panics with "concurrent write to websocket connection"

### Pitfall 3: Goroutine Leak on Client Disconnect
**What goes wrong:** readPump or writePump goroutine keeps running after client disconnects
**Why it happens:** Missing cleanup when connection closes; no unregister from Hub
**How to avoid:** readPump defers `hub.unregister <- client` and `conn.Close()`; writePump defers `conn.Close()` and stops ticker
**Warning signs:** Growing goroutine count over time, memory leak

### Pitfall 4: Tailwind v4 Requires Input CSS File
**What goes wrong:** Running `tailwindcss -o output.css` without `-i input.css` fails in v4
**Why it happens:** v4 changed from v3 -- config file approach no longer works, input CSS file with `@import "tailwindcss"` is mandatory
**How to avoid:** Create `input.css` with `@import "tailwindcss"; @source "../templates/**/*.html";` and always use `-i` flag
**Warning signs:** Empty or missing output.css, no utility classes generated

### Pitfall 5: Tailwind Not Scanning Go Template Files
**What goes wrong:** Tailwind doesn't detect utility classes in `.html` Go template files
**Why it happens:** Tailwind v4 auto-detects from the CSS file's directory but may not traverse to template directories
**How to avoid:** Use explicit `@source` directive in input.css: `@source "../templates/**/*.html";`
**Warning signs:** Classes present in HTML but not in generated CSS

### Pitfall 6: go:embed Path Must Be Relative
**What goes wrong:** `//go:embed` directive fails to compile
**Why it happens:** Paths in `//go:embed` must be relative to the Go source file, cannot use `..` or absolute paths
**How to avoid:** Place the `//go:embed` directive in the `dashboard/` package, with static files as subdirectories of `dashboard/`
**Warning signs:** Compile error: "pattern ... cannot match directory outside package"

### Pitfall 7: Dashboard Panic Crashes Trading Bot
**What goes wrong:** Unhandled panic in WebSocket handler or Hub goroutine kills entire process
**Why it happens:** Go panics propagate up the goroutine stack and crash the process
**How to avoid:** (1) Gin's `Recovery()` middleware catches handler panics. (2) Wrap Hub.Run() and Ticker.Run() in `defer func() { if r := recover(); r != nil { log.Error(...) } }()`
**Warning signs:** Bot process exits with panic stack trace from dashboard code

### Pitfall 8: Browser WebSocket Reconnection
**What goes wrong:** Dashboard shows stale data after temporary network interruption
**Why it happens:** Browser WebSocket does not auto-reconnect
**How to avoid:** In `ws.onclose` handler, set `connected = false` and call `setTimeout(() => connectWebSocket(), 3000)` with backoff
**Warning signs:** Dashboard shows "disconnected" and never recovers

## go:embed vs Filesystem: Decision

The blocker/concern mentions needing a decision on `go:embed` vs filesystem serving toggle mechanism.

**Recommendation: Use go:embed with development override via build tag or env var.**

During development, the developer can either:
1. **Rebuild on change** (simplest): `go run ./cmd/bot` after template changes
2. **Build tag approach**: Use `//go:build dev` to swap embed.FS for os.DirFS at compile time
3. **Air or similar** live reload tool for Go

**For Phase 1, start with go:embed only (option 1).** The templates will be iterated during this phase. Rebuilding is fast (Go compiles in seconds). A development toggle can be added later if the rebuild cycle becomes painful. This avoids adding complexity before it is needed.

## Open Questions

1. **Dashboard polling interval**
   - What we know: Ticker polls services at configurable intervals. Alpaca has rate limits.
   - What's unclear: Optimal interval for real-time feel without hammering Alpaca API
   - Recommendation: Default 2 seconds for account/positions (matches Alpaca's update frequency), 5 seconds for activity feed. Make configurable via env var `DASHBOARD_POLL_INTERVAL`.

2. **Activity feed data source**
   - What we know: ActivityLogger writes to JSON files. Existing REST endpoint reads from ActivityLogger in-memory state.
   - What's unclear: Whether to read activity from in-memory ActivityLogger or parse the JSON files
   - Recommendation: Read from in-memory ActivityLogger via `GetCurrentLog()` -- already in memory, no disk I/O. For historical activity, read from JSON files.

3. **Existing `/dashboard` static route conflict**
   - What we know: main.go line 229 has `router.Static("/dashboard", "./web")` serving a `./web` directory
   - What's unclear: Whether this `./web` directory exists or has content
   - Recommendation: Replace this route with the new embedded dashboard. The new `dashboard.Register()` function will own the `/dashboard` route.
